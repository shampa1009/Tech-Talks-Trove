𝐎𝐮𝐫 𝐒𝐭𝐚𝐫 𝐃𝐞𝐯𝐞𝐥𝐨𝐩𝐞𝐫 𝐜𝐚𝐧 𝐛𝐞 𝐨𝐮𝐫 𝐛𝐢𝐠𝐠𝐞𝐬𝐭 𝐒𝐲𝐬𝐭𝐞𝐦 𝐑𝐢𝐬𝐤.

আমরা প্রায় প্রতিটা টিমেই এমন একজনকে দেখি, ধরা যাক তার নাম Rahim.

Rahim সব জানে.
Rahim সব ঠিক করতে পারে.
Production down? call Rahim.
Deadline risk? Rahim manages.

Leadership ভাবে Rahim হলো আমাদের team-এর backbone. Release বাঁচানোর hero.

But এখানে একটা silent problem grow করে...

Knowledge spread না হয়ে concentrate হয়ে যায় Rahim-এর কাছে.
Team-এর অন্যরা ধীরে ধীরে depend করতে শুরু কর"Rahim তো আছেই."
Documentation হয় না — কারণ Rahim already জানে.
Codebase এমনভাবে grow করে যেখানে context শুধু Rahim-এর মাথায় থাকে.

Then one day... Rahim resign করে.
Suddenly আমরা বুঝতে পারি —
Rahim আমাদের team-কে scale করেনি.
Rahim unknowingly আমাদের system-এর দুর্বলতা cover করছিল.

👉 The real anti-pattern is NOT the genius developer.
👉 The real problem is organizational culture that rewards heroics instead of building sustainable systems.

অনেক organization crisis solve করা মানুষকে celebrate করে, কিন্তু crisis prevent করার system build করে না.

High-performing teams hero dependent না.
তারা এমন process & knowledge-sharing culture build করে যেখানে কেউ single point of failure না.

If we notice this pattern in our team, maybe we should start here:
✅ Strategic pair programming শুরু করা
✅ Ownership rotate করা
✅ Documentation কে optional না, job responsibility বানানো
✅ Knowledge sharing কে performance metric বানানো

শেষ প্রশ্নটা খুব important —
❝ How do we retain Rahim? ❞ না…

✅ ❝ Why does our system need a Rahim to survive? ❞
